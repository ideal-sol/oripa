## 2. 既存仕様として維持する不変条件

以下は既存の確定仕様を変更せず、V2でも維持する。

1. 抽選、確率、ポイント消費ロジックはLaravelだけに置く。
2. 抽選乱数はCSPRNGを使用する。
3. 確率はppm整数で管理する。
4. 公開済み確率Versionはimmutableとする。
5. `draw_sequence_number`はガチャ行のDB Lock下で採番する。
6. 抽選、ポイント消費、在庫、`sold_count`、`won_count`、抽選結果、ポイント台帳、獲得景品は同一DB Transaction内で整合させる。
7. paidポイントは無期限とし、`expire_at = null`とする。
8. freeポイントだけを失効対象とする。
9. ポイント消費順は次のとおりとする。
   - freeを優先
   - free内では失効期限が近い順
   - 同一期限では付与日時が古い順
   - その後paidを付与日時が古い順に消費
10. WalletとPoint Lotをマイナスにしない。
11. 配送状態と追跡番号は景品単位で管理する。
12. 通常返金は、対象Payment由来のポイントが全額未使用の場合だけ可能とする。
13. 1ptでも使用済みの場合、通常返金は不可とする。
14. チャージバックでは、現在保有するポイントから取消する。
15. チャージバックの取消で不足が発生しても、WalletやLotを負数にしない。
16. 不足額だけを表すPoint Ledgerは作成しない。
17. 抽選履歴、抽選連番、`sold_count`、`won_count`は返金・チャージバックでも巻き戻さない。
18. 通知失敗で返金・チャージバック等の本体処理をRollbackしない。
19. 業務日付はAsia/Tokyo基準とする。
20. QA抽選も模擬処理ではなく、通常のポイント・在庫・景品データを実際に更新する。

## 3. V2の最終決定一覧

| 項目 | V2確定方針 |
|---|---|
| サイト間のDB | 完全分離 |
| `tenant_id` | 使用しない |
| DB | PostgreSQL |
| Redis | Cache / Queue専用。残高の正本にしない |
| 内部Primary Key | `bigint identity` |
| API公開ID | `public_id uuid`、APIではopaque string |
| 金額・ポイント | `bigint`整数 |
| 小数・float | 金額・ポイントには使用禁止 |
| 通貨 | V2.0はJPY |
| paidポイント価値 | **1 paid point = 1 JPY** |
| 購入Bonus | 必ずfreeポイントとして付与 |
| paidポイント生成元 | 成功済みの実決済を原則とする |
| paidポイント期限 | 無期限 |
| freeポイント期限 | 必須 |
| 現在残高 | Wallet＋Point Lotを同一Transactionで更新 |
| 履歴 | immutableなPoint Ledger Entry |
| 操作単位 | Point Operation |
| 決済とポイント | Payment成功確定後に一度だけ付与 |
| Webhook | Provider Eventを独立保存 |
| 返金・CB | Payment Adjustmentを1:Nで保存 |
| 通常返金 | V2.0では全額返金のみ |
| 部分返金 | Schemaは拡張可能、APIは無効 |
| CB取消 | Schema対応、V2.0自動復元は禁止 |
| 外部通信 | DB Transaction外で実行 |
| 通知 | Transactional Outbox |
| 日次残高 | Ledger cutoffで正確に再構築 |
| QA | Point Operation等にQA識別を保存 |
| 削除 | 金融・ポイント履歴の物理削除禁止 |
| Mock決済 | Productionで起動禁止 |

### 3.1 V2新規確定: paidポイントの価値

V2.0では次を製品不変条件とする。

```text
1 paid point = 1 JPY
```

例:

```text
決済金額 1,000円
paidポイント 1,000pt
free Bonus 200pt
合計付与 1,200pt
```

次は禁止する。

```text
決済金額 1,000円
paidポイント 1,200pt
```

追加分は必ずfreeポイントとして保存する。

これにより、paidポイント残高と対価額の対応関係を明確にする。ただし、法的な未使用残高の算定方法や届出・供託等は専門家確認を必須とする。

## 4. 各サイトの完全分離

各導入サイトは、次をすべて個別に持つ。

```text
顧客A
├── 顧客A専用Laravel
├── 顧客A専用PostgreSQL
├── 顧客A専用Redis
├── 顧客A専用Queue
├── 顧客A専用Scheduler
├── 顧客A専用Storage
├── 顧客A専用決済Provider設定
├── 顧客A専用Secret
├── 顧客A専用Backup
└── 顧客A専用監視・ログ
```

別サイトも同様に、完全に独立した環境を持つ。

V2では次を行わない。

- 共有PostgreSQL
- 共有Schema
- 全テーブルへの`tenant_id`
- サイトをまたぐUser
- サイトをまたぐWallet
- サイトをまたぐPayment
- サイトをまたぐPoint Ledger
- サイト間の分散Transaction

共通化するのはソースコード、Migration、Docker Image、API仕様、Storefront Clientだけである。

## 5. PostgreSQL共通規約

### 5.1 Primary Keyと公開ID

内部DB Relationは次を標準とする。

```sql
id bigint generated by default as identity primary key
```

Public APIへ公開する主要Resourceは次も持つ。

```sql
public_id uuid not null unique
```

`public_id`はApplication側でUUIDv7を生成する。

対象例:

- users
- payments
- payment_adjustments
- point_operations
- point_adjustments
- draw_requests
- user_prizes
- shipping_requests
- export_jobs

APIは内部`id`を契約として公開しない。

### 5.2 日時

DB Timestampは次を標準とする。

```text
timestamptz
UTC保存
```

業務日付は次を使用する。

```text
date
Asia/Tokyoで算出
```

例:

```text
occurred_at       timestamptz
business_date     date
```

### 5.3 金額とポイント

```text
金額              bigint
ポイント          bigint
通貨              char(3)
```

V2.0の通貨は`JPY`だけを許可する。

次は禁止する。

- float
- double
- 小数点付き金額
- JavaScript Numberだけに依存した大きな値の演算

API上も整数として扱い、上限をApplication Validationで設定する。

### 5.4 状態値

PostgreSQL native enumは使用せず、次を採用する。

```text
varchar
＋
CHECK constraint
```

理由:

- Package Version更新時の状態追加を扱いやすくする
- MigrationのRollbackや展開を単純にする
- 顧客別環境への段階展開をしやすくする

### 5.5 JSONB

JSONBは次だけに使用する。

- Provider固有の非中核metadata
- 監査用の補助情報
- Outbox payload
- Redaction済みの外部応答

次をJSONBだけで管理してはいけない。

- Payment status
- 金額
- paid/freeポイント数
- Provider event ID
- Adjustment type
- 返金・CB status
- Point Ledger amount
- 在庫数
- 失効日時
- 検索・集計に必要な項目

### 5.6 Foreign Keyと削除

金融・ポイント系Foreign Keyでは原則として次を使用する。

```text
ON DELETE RESTRICT
```

または、削除対象のUserを残して匿名化する。

次は禁止する。

```text
users削除
  ↓ CASCADE
payments / point_lots / point_ledger_entriesを削除
```

実決済またはポイント履歴を持つUserは物理削除せず、個人情報だけを匿名化する。

## 6. データの責任分担

ポイントは3層で管理する。

```text
Wallet
├── 現在残高の集約
├── User単位のLock起点
└── 高速な残高表示

Point Lot
├── paid/free区分
├── 付与元
├── 残量
├── 予約量
└── 失効期限

Point Ledger Entry
├── immutable
├── すべての増減
├── 操作後残高
└── 過去時点の再構築・監査
```

### 6.1 正本の考え方

単一テーブルだけを「唯一の正本」としない。

用途別に次を正とする。

| 用途 | 正とするデータ |
|---|---|
| 現在利用可能残高 | LockされたWallet＋Point Lot |
| 消費順 | Point Lot |
| 増減履歴 | Point Ledger Entry |
| 一連の業務操作 | Point Operation |
| 過去時点の再構築 | Point Ledger Entry |
| 日次・基準日残高 | Ledgerから生成したSnapshot |
| 決済成立 | Payment＋Provider Event |
| 返金・CB履歴 | Payment Adjustment |

Wallet、Lot、Ledgerは必ず同じDB Transaction内で更新する。

| `updated_at` | timestamptz | |


## 7.3 `point_lots`

付与単位のポイント残高。

主要Column:

| Column | 型 | 説明 |
|---|---|---|
| `id` | bigint | PK |
| `user_id` | bigint | FK |
| `grant_operation_id` | bigint | FK |
| `point_type` | varchar | paid / free |
| `granted_amount` | bigint | 初期付与量 |
| `remaining_amount` | bigint | 現在残量 |
| `reserved_amount` | bigint | 返金等の予約量 |
| `granted_at` | timestamptz | 付与日時 |
| `expire_at` | timestamptz nullable | freeのみ必須 |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | |

Constraint:

```text
granted_amount > 0
remaining_amount >= 0
reserved_amount >= 0
reserved_amount <= remaining_amount
remaining_amount <= granted_amount

paid  → expire_at IS NULL
free  → expire_at IS NOT NULL
```

paidポイントを生成できる通常Sourceは、成功済みPaymentだけとする。

現金対価を伴わない次のポイントは原則freeとする。

- 購入Bonus
- 紹介
- LINE友達追加
- キャンペーン
- 補償
- point_back
- 最低保証
- 景品ポイント交換
- 運営付与

管理者によるpaid付与は例外操作とし、権限・理由・監査を必須にする。最終権限は認証・権限基準で確定する。

## 7.7 `point_balance_snapshots`

### V2新規確定: 過去残高の正確なCutoff

現在の「00:10に現在残高を取得し、前日の日付へ保存する」方式は、V2の正式Snapshot方式として使用しない。

V2では、前日終了時点のLedgerを基準に正確に再構築する。

```text
Snapshot Date: 2026-07-21
Cutoff:        2026-07-22 00:00:00 Asia/Tokyo
生成時刻:      2026-07-22 00:10以降
```

`occurred_at < cutoff`のLedger Entryだけを使う。

主要Column:

| Column | 型 | 説明 |
|---|---|---|
| `id` | bigint | PK |
| `snapshot_date` | date | UNIQUE |
| `source_cutoff_at` | timestamptz | 対象Cutoff |
| `calculation_method` | varchar | ledger_cutoff |
| `opening_paid_balance` | bigint | |
| `opening_free_balance` | bigint | |
| `granted_paid_amount` | bigint | |
| `granted_free_amount` | bigint | |
| `consumed_paid_amount` | bigint | |
| `consumed_free_amount` | bigint | |
| `expired_free_amount` | bigint | |
| `reversed_paid_amount` | bigint | |
| `reversed_free_amount` | bigint | |
| `closing_paid_balance` | bigint | |
| `closing_free_balance` | bigint | |
| `paid_reserved_balance` | bigint | |
| `free_reserved_balance` | bigint | |
| `user_count` | bigint | |
| `open_lot_count` | bigint | |
| `is_base_date` | boolean | 3/31、9/30 |
| `generated_at` | timestamptz | |
| `generation_run_id` | uuid | |
| `checksum` | varchar | |

Rule:

- `snapshot_date`は一意。
- 同日再生成はVersion履歴または監査記録を残す。
- 任意の過去日へ現在残高を保存するCommandは禁止する。
- 過去日再生成はLedger cutoff方式だけを許可する。
- 3月31日と9月30日は`is_base_date = true`とする。
- 法務・会計向け出力ではpaidとfreeを混在させない。

## 8. ポイント消費処理

標準Transaction:

```text
1. Idempotency / Draw Requestを確認
2. User WalletをSELECT ... FOR UPDATE
3. 利用可能残高を確認
4. free Lotを順序固定でSELECT ... FOR UPDATE
5. 不足分だけpaid Lotを順序固定でSELECT ... FOR UPDATE
6. Point Operation作成
7. Lotを減算
8. Walletを減算
9. Ledger Entry作成
10. 抽選・在庫・結果・獲得景品を同一Transactionで更新
11. Commit
12. 通知をOutboxから処理
```

free Lot順序:

```sql
ORDER BY expire_at ASC, granted_at ASC, id ASC
```

paid Lot順序:

```sql
ORDER BY granted_at ASC, id ASC
```

利用可能量:

```text
remaining_amount - reserved_amount
```

次は禁止する。

- Redis残高だけで消費可否を決定
- WalletをLockせずにLotを消費
- `SKIP LOCKED`でLock中のLotを飛ばす
- 外部APIをTransaction内で呼ぶ
- Frontendで消費順を決定
- Point Ledgerだけを作成しWallet/Lotを後から更新

## 9. freeポイント失効

失効処理:

```text
1. expire_at <= cutoffのfree Lotを対象
2. User単位でWalletをLock
3. LotをLock
4. active ReservationがあるLotは失効対象外または要手動解決
5. Point Operationを作成
6. remaining_amountを0へ
7. Wallet free_balanceを減算
8. Ledger Entryを作成
9. Commit
```

Rule:

- paid Lotは失効させない。
- 失効額は負数Ledger Entryとして残す。
- 失効通知はOutboxでCommit後に送る。
- 同一Lotを二重失効しないUnique business keyを持たせる。

## 14. 通常返金

### 14.1 V2.0の返金条件

次をすべて満たす場合だけ通常返金可能。

1. Paymentが`succeeded`
2. 対象Paymentに成功済み返金・CBがない
3. 対象Payment由来paidポイントが全額未使用
4. 対象Payment由来free Bonusが全額未使用
5. 対象Lotが失効・取消・予約済みではない
6. Provider側で返金可能期間内
7. 金額はPayment全額
8. Idempotency-Keyが有効

対象ポイントが1ptでも使用済みなら返金不可。

### 14.2 返金Saga

Provider呼び出しをDB Transaction内に含めない。

#### Phase A: 予約

```text
1. Idempotency Recordを作成・Lock
2. PaymentをLock
3. WalletをLock
4. Payment由来LotをLock
5. 全額未使用を再検証
6. Payment Adjustmentをrequestedで作成
7. Point Lot Reservationを作成
8. Lot / Walletのreserved amountを加算
9. Adjustmentをpoints_reservedへ
10. Provider Refund CommandをOutboxへ
11. Commit
```

予約中のポイントはUserが消費できない。

#### Phase B: Provider操作

WorkerがProviderへ返金要求を送る。

- Provider idempotency keyを固定
- Timeout時は結果不明として記録
- 結果不明時にReservationを解放しない
- Provider照合を行う

#### Phase C: 成功確定

```text
1. Adjustment / Payment / Wallet / LotをLock
2. 予約量をPoint残高から取消
3. Point OperationとLedgerを作成
4. Reservationをconsumedへ
5. Wallet reservedを減算
6. Adjustmentをsucceededへ
7. Status HistoryとOutboxを作成
8. Commit
```

#### Phase D: 明確な失敗

```text
1. Reservationをreleasedへ
2. Wallet / Lot reservedを減算
3. Adjustmentをfailedへ
4. Commit
```

通知失敗で返金結果をRollbackしない。

## 15. チャージバック

チャージバックはProvider Eventを起点とする。

対象Payment由来ポイントが使用済みでも処理する。

### 15.1 取消順序

既存仕様を維持し、次の順で取消する。

#### Step 1: 購入paid分

1. 対象Paymentのpaid Lot
2. その他のpaid LotをFIFO

#### Step 2: 購入free Bonus分

1. 対象Paymentのfree Lot
2. その他のfree Lotを通常のfree消費順

#### Step 3: paid不足分

残っているfree Lotから取消する。

#### Step 4: Shortfall

残額を`payment_adjustment_point_impacts`へ保存する。

絶対禁止:

```text
free Bonus分をpaid Lotから取消する
Walletを負数にする
Lotを負数にする
Shortfall用の負残高Ledgerを作る
```

### 15.2 景品対応

- 未発送景品はhold
- 発送済み・配送済みは返送依頼
- 抽選履歴は削除しない
- 抽選連番を戻さない
- `sold_count`、`won_count`を戻さない
- 景品Actionを個別Recordで残す
- メールはCommit後にOutboxで送信

### 15.3 アカウント制限

Shortfallや未返送景品がある場合のアカウント制限は、認証・権限・不正対策設計で確定する。

DBはAdjustment、Shortfall、Prize Actionを保持し、制限判断に利用できる状態にする。

## 16. チャージバック取消

Schemaでは次を許容する。

```text

```

ただし、V2.0では自動的なポイント・景品状態復元を禁止する。

理由:

- 元のCB取消が対象Payment以外のLotにも及ぶ可能性がある

自動復元を追加する場合は、別ADRと専用Testを必須とする。

## 19. TransactionとLock

### 19.1 Isolation Level

通常処理はPostgreSQLの`READ COMMITTED`を使用し、重要行に明示的な`SELECT ... FOR UPDATE`を行う。

全抽選を無条件に`SERIALIZABLE`へ変更しない。

DeadlockまたはSerialization Failure時は、同一Idempotency-Keyで最大3回まで短いJitter付きで再試行する。

対象SQLSTATE:

```text
40001 serialization_failure
40P01 deadlock_detected
```

### 19.2 Lock順序

#### 抽選

```text
1. Idempotency / Draw Request
2. Gacha
3. Wallet
4. Point Lots
5. Prize / Inventory rows
6. Draw Results / User Prizes
```

#### Payment成功

```text
1. Provider Event
2. Payment
3. Wallet
4. Point Lots / Ledger
```

#### 通常返金

```text
1. Idempotency / Adjustment
2. Payment
3. Wallet
4. Payment-origin Lots
5. Reservations
```

#### チャージバック

```text
1. Provider Event
2. Payment
3. Adjustment
4. Wallet
5. Point Lots
6. User Prizes
```

同一種類の複数Rowは必ずID昇順でLockする。

### 19.3 Transaction内で禁止する処理

- 決済Provider API
- Mail送信
- Discord通知
- SMS送信
- S3 Upload
- 長時間のCSV生成
- 外部HTTP
- User入力待ち


Foreign Key側の検索Columnへ必要なIndexを付与する。

Indexは本番相当データで`EXPLAIN ANALYZE`を確認し、利用実績に基づいて追加・削除する。

初期段階からすべての組み合わせへIndexを作らない。

## 25. User削除と匿名化

実決済・ポイント・抽選・配送履歴を持つUserは物理削除しない。

退会・削除依頼時:

```text
1. Loginを無効化
2. 個人情報を匿名化または法定期間に従って隔離
3. User ID自体は保持
4. Payment / Point / Draw / Shipping relationを保持
5. Audit Logを保存
```

具体的な保持期間と匿名化対象は、プライバシー・法務設計で確定する。

## 27. V1からV2へのMigration

### 27.1 前提

現時点では本番運用・本番決済が始まっていないため、V2 Productionは新しい空DBから開始する。

V1のmock決済、テストUser、テストPoint残高はV2 Productionへ移行しない。

移行候補:

- カテゴリ
- タグ
- 景品Master
- 確率Version
- 演出素材
- コンテンツ
- 運用設定

明示確認後にImportする。

### 27.2 実施順

```text
1. V1未コミット差分を保全
2. v1-before-productization Tag
3. V1 Schema Dump
4. V2 Migrationをfresh実行
5. Master Data Importerを作成
6. Test環境で件数・Checksum確認
7. V2 Seed / Import
8. Point・Paymentは初期0件で開始
9. 本番決済を接続
10. 開始時点Snapshotを保存
```

### 27.3 V1 Migration

V1 Migrationを削除・書換えしない。

V2は別のbaseline Migration群を作成する。

V2公開後はforward-only Migrationとする。

## 33. 現行設計からの主要変更

| 現行・暫定 | V2確定 |
|---|---|
| `payment_reversals.payment_id UNIQUE` | `payment_adjustments` 1:N |
| Paymentに`webhook_event_id` | `payment_provider_events` 1:N |
| Payment statusに返金・CB | Payment lifecycleとAdjustmentを分離 |
| LotとLedger中心 | Point Operation親Recordを追加 |
| 返金時に外部結果待ち中の保護が弱い | Point Lot Reservation |
| Snapshotを実行時残高から作成 | Ledger cutoffで過去時点を再構築 |
| 任意日へ現在残高を保存可能 | 禁止 |
| paidと円価値の関係が暗黙 | 1 paid point = 1 JPY |
| 公開済み購入Planの編集余地 | Published後immutable |
| Mock決済を設定で利用 | Productionで強制停止 |
| 金融RecordのCascade余地 | RESTRICT＋User匿名化 |
| 単一Reversal | Refund / CB / CB取消をAdjustmentで履歴化 |
| 通知を個別にafter commit | Transactional Outbox |
| 手動照合 | 日次Reconciliationを標準化 |
